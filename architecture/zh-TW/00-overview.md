# 第一部分 — 導航

## §1 全局架構圖

Claude Code 的強大不是來自某個殺手鐧功能，而是**六大支柱協同運作**——每一個都在放大其他支柱的效果：

```
                        ┌─────────────────────────┐
                        │     System Prompt        │
                        │  (身份 + 規則 +           │
                        │   42+ 工具描述)           │
                        └────────────┬────────────┘
                                     │
                  ┌──────────────────┼──────────────────┐
                  │                  │                  │
         ┌───────▼────────┐ ┌──────▼───────┐ ┌───────▼────────┐
         │  工具系統       │ │  查詢循環    │ │    上下文      │
         │  (42+ 工具，    │ │  (12 步      │ │  管理          │
         │   每個 30+方法) │ │   狀態機)    │ │  (4 層壓縮)    │
         └───────┬────────┘ └──────┬───────┘ └───────┬────────┘
                  │                  │                  │
                  └──────────────────┼──────────────────┘
                                     │
                  ┌──────────────────┼──────────────────┐
                  │                  │                  │
         ┌───────▼────────┐ ┌──────▼───────┐ ┌───────▼────────┐
         │  權限與安全     │ │  多 Agent    │ │  Skill &       │
         │  (7 層縱深防禦) │ │  集群        │ │  Plugin        │
         │                │ │  (3 後端，   │ │  (6 源，       │
         │                │ │   7 種任務)  │ │   MCP 協議)    │
         └────────────────┘ └──────────────┘ └────────────────┘
```

**一句話數據流：**

```
用戶輸入 → CLI (cli.tsx) → QueryEngine.query() → System Prompt 組裝
  → 流式 API 調用 → 工具提取 → 權限檢查 → 工具執行
  → 結果注入 → 繼續/結束決策 → 下一次迭代或響應
```

**六大設計哲學：**

| # | 哲學 | 一句話 |
|---|------|--------|
| 1 | **LLM = 大腦，Harness = 身體** | LLM 負責推理；Harness 負責感知、行動、記憶和約束 |
| 2 | **工具 = 能力邊界** | Claude 只能做註冊工具允許的事情 |
| 3 | **上下文是最稀缺的資源** | 200K token 聽起來很多，一箇中等項目就能填滿 |
| 4 | **零信任安全** | 每一次工具調用都經過多層權限檢查 |
| 5 | **循環直到完成** | 不是請求-響應，而是 `調用 → 工具 → 結果 → 重複` |
| 6 | **可擴展性決定上限** | Skill、Plugin、MCP 讓能力無限延伸 |

## §4 Harness：大腦周圍的身體

LLM 是大腦，*Harness* 是其他一切——賦予大腦感知、行動、記憶和安全的運行時基礎設施：

| 能力 | 實現 | 大腦看到的 |
|------|------|-----------|
| **感知文件** | FileReadTool, GlobTool, GrepTool | "我能看到代碼庫" |
| **修改代碼** | FileEditTool, FileWriteTool | "我能修改東西" |
| **執行命令** | BashTool（沙箱化） | "我能運行和測試" |
| **搜索網絡** | WebSearchTool, WebFetchTool | "我能查資料" |
| **與用戶交互** | AskUserQuestionTool, 權限對話框 | "我能提問" |
| **分派工作** | AgentTool, TeamCreate, SendMessage | "我能委派任務" |
| **記憶** | CLAUDE.md, memdir, auto-memory | "我有長期記憶" |
| **擴展能力** | MCPTool, SkillTool, Plugin 系統 | "我能學新技能" |

**System Prompt 的分層構建：**

```
System Prompt 構建（按 Token 優先級排序）：
  1. 核心身份與規則    ← 我是誰，能/不能做什麼
  2. 工具描述（42+）   ← ~10K+ tokens 的能力定義
  3. Git 狀態與項目上下文 ← 當前分支、最近提交
  4. CLAUDE.md 內容    ← 項目級指令
  5. 用戶/企業規則     ← 偏好、組織策略
  6. 動態附件          ← 技能發現、記憶注入、MCP 資源
```

## §5 核心循環：12 步狀態機

Claude Code 不是"發送提示，得到回答"，而是一個 **while(true) 循環**——將單次調用的 LLM 變成迭代式 Agent 的核心放大器：

| 步驟 | 名稱 | 目的 |
|------|------|------|
| 1 | `applyToolResultBudget` | 裁剪超大工具結果 |
| 2 | `snipCompact` | 截斷最舊對話輪次 |
| 3 | `microcompact` | 清理過期工具輸出（時間衰減） |
| 4 | `contextCollapse` | 摺疊冗餘上下文 |
| 5 | `autocompact` | AI 生成對話摘要 |
| 6 | `tokenWarningState` | 預算檢查與用戶告警 |
| 7 | `callModel` | 流式 API 調用 |
| 8 | `StreamingToolExecutor` | *邊流式輸出邊執行*工具 |
| 9 | `stopHooks` | 評估停止條件 |
| 10 | `tokenBudget check` | 硬預算限制 |
| 11 | `getAttachmentMessages` | 注入記憶、技能發現 |
| 12 | `state transition` | 更新狀態，繼續循環 |

**錯誤恢復策略矩陣：** 429 → 指數退避；529 → 降級到 Sonnet；prompt-too-long → 緊急壓縮；max_output_tokens → 自動提升 8K→64K。

## §6 工具系統：契約式能力邊界

每個工具實現 **30+ 方法的完整契約**——身份、執行、安全、併發、UI 渲染。`buildTool` 工廠提供 fail-closed 默認值（`isConcurrencySafe: false`，`isReadOnly: false`）。**三層工具過濾**確保模型只看到當前上下文適用的工具，`assembleToolPool()` 的穩定排序保證 Prompt Cache 鍵穩定。

**BashTool** 是最複雜的工具（~156K 代碼），採用 6 層縱深防禦——從安全包裝器剝離到 tree-sitter AST 解析再到 macOS Seatbelt OS 級沙箱。

## §7 上下文管理：最稀缺的資源

**四層壓縮保持系統活力：** Layer 0（API 側自動清除）→ Layer 1 微壓縮（時間衰減清理舊結果）→ Layer 2 截斷（丟棄最舊輪次）→ Layer 3 自動壓縮（AI 生成摘要，帶電路斷路器）→ Layer 4 緊急壓縮（僅在 API 返回 prompt-too-long 時觸發）。

**Token 估算三級精度：** 粗略（`bytes/4`，零成本）→ 代理（Haiku `usage.input_tokens`，便宜）→ 精確（`countTokens` API，準確但慢）。

**Prompt Cache 三層策略** + **Beta header 鎖存**保持緩存鍵穩定，避免 12 倍成本懲罰。

> → 深度解析: [EP10: 上下文組裝](10-context-assembly.md) | [EP11: 壓縮系統](11-compact-system.md)

## §8 權限與安全：地基而非天花板

安全採用**縱深防禦**——每層假設上一層已被繞過：工具級驗證 → 7 源權限規則合併 → Bash AST 安全分析（23 種注入檢測 + fail-closed）→ 投機性 YOLO 分類器（2 秒超時）→ 用戶確認對話框 → OS 級沙箱。

**設計原則：** 安全相關 **fail-closed**（未知 → 拒絕）；可用性相關 **fail-open**（策略服務宕機 → 允許所有，使用本地緩存）。

## §11 七個值得研究的工程實踐

### 11.1 編譯時死代碼消除

`feature('VOICE_MODE')` 在編譯時被替換為 `true`/`false`，`false` 分支被 Bun 打包器**完全移除**。三種條件加載層級共存：編譯時 DCE → 運行時環境檢查 → 運行時能力檢測。

### 11.2 循環依賴消除：葉模塊模式

`bootstrap/state.ts`（~55KB）被所有模塊依賴，解決方案是讓它成為**依賴圖的葉節點**——禁止從 `src/` 導入任何東西。這不是建議，而是由自定義 ESLint 規則（`custom-rules/bootstrap-isolation`）**強制執行**的架構約束。

### 11.3 35 行 Store 替代 Redux

35 行 `createStore` 函數提供 `getState`/`setState`/`subscribe` 接口，配合 React 19 的 `useSyncExternalStore` 完成狀態管理。`Object.is` 引用相等跳過避免不必要的重渲染。依賴成本：0 字節。

### 11.4 Fork Ink：在終端中構建瀏覽器

原版 Ink 的 LegacyRoot → `ConcurrentRoot`（React 19），無事件系統 → W3C 捕獲/冒泡模型，單緩衝 → 雙緩衝 + ANSI diff。渲染管線：`stdin → 按鍵解析 → DOM 事件分發 → React 更新 → Yoga 佈局 → 屏幕緩衝 → ANSI diff → stdout`（16ms 節流）。

### 11.5 Vim 模式：教科書級狀態機

四態有限狀態機：Normal → Insert / Operator Pending / Find Pending。每個狀態有明確的進入/退出動作和有限的狀態轉換集——行為可預測、可測試、可擴展。

### 11.6 Import-Gap 並行化

在 `import` 語句之間啟動異步操作，利用 ES 模塊評估的"空閒窗口"（~135ms）作為免費並行時間。無需 `Promise.all`，只需把異步啟動放在 import 之間。

### 11.7 Fail-Open 與 Fail-Closed 的工程抉擇

**Fail-Closed（安全優先）：** Bash 解析錯誤 → 拒絕執行；權限超時 → 默認拒絕。**Fail-Open（可用性優先）：** 策略加載失敗 → 允許所有；遠程配置超時 → 用本地緩存；GrowthBook 不可用 → 用默認值。邊界清晰：涉及用戶數據/系統訪問用 fail-closed，涉及配置/外部服務用 fail-open。

# 第四部分 — 獨特視角

## §12 第一人稱視角：我是如何運作的

### 12.1 我的思維循環

每次工具調用都是一次完整的 API 往返。你感受到的"思考 → 行動 → 觀察"循環，在底層是：`API 響應（tool_use）→ Harness 執行 → 結果注入 → 下一次 API 調用`。修復一個 bug 可能需要 5 次 API 調用：讀文件 → 搜索 → 確認 → 編輯 → 測試。

### 12.2 我的並行能力

我可以在一次 API 響應中調用多個工具，Harness 通過 `StreamingToolExecutor` 並行執行所有 `isConcurrencySafe: true` 的工具。讀操作並行，寫操作串行。

### 12.3 我的上下文感知

我通過六個通道感知世界：System Prompt（我的"DNA"）、CLAUDE.md（我的"項目記憶"）、對話歷史（我的"短期記憶"）、工具結果（我的"感官"）、Memory/memdir（我的"長期記憶"）、動態附件（我的"外部知識"）。

### 12.4 我的侷限性

有限的上下文窗口 → 4 層壓縮自動管理；不能直接執行代碼 → BashTool 代勞；可能輸出不安全命令 → 6 層安全管道攔截；單次輸出有限 → `max_output_tokens` 自動提升；可能產生幻覺 → 工具結果提供事實反饋；API 不穩定 → 重試+降級+錯誤恢復；單一上下文不夠 → 多 Agent 分治。

### 12.5 我為什麼能高效工作

42+ 種工具覆蓋幾乎所有場景；可以迭代試錯；可以並行化；有跨會話記憶；被安全約束（用戶放心給權限）；能自我修復（錯誤恢復機制）。

### 詳解：鎖存模式（Pattern 9）

**場景：** API beta header 影響 Prompt Cache 鍵。**問題：** 中途切換 header 導致所有緩存失效（12 倍成本增加）。**方案：** 三態鎖存——`null`（未知）→ `true`（鎖定）→ 永遠保持 `true`。**效果：** 整個會話緩存鍵穩定，一個決策，無翻轉。

### 詳解：閉包工廠（Pattern 14）

**場景：** LSP 服務器管理器需要私有狀態 + 公共接口。**問題：** Class 帶來繼承誘惑和 `this` 綁定問題。**方案：** 工廠函數返回閉包對象，私有狀態在閉包作用域中。代際計數器（`++generationCounter`）防止過期的異步操作覆蓋新狀態。**效果：** 比 class 更乾淨的 API，無 `this` 困惑。
